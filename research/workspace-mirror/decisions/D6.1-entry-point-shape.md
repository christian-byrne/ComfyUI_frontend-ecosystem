`app.registerExtension` vs `defineExtension` vs Hybrid

Two independent questions:
1. **Call site** — what does an extension author actually write?
2. **Dispatch mechanism** — if the call site is shared, how does the host know which version to invoke?

---

*Axis 1: Call site*

*Option 1*: `app.registerExtension({...})` (no new import)

Same call site for v1 and v2. Host detects version at registration or dispatch time (see Axis 2).

```ts
app.registerExtension({ name: 'MyExt', ... })
```

- Pros
    - Zero call-site change for any migrating author
    - All 119 starred repos already know this function
    - No new package import required
- Cons
    - `window.app` must be initialized before registration fires
    - Cannot warn "stop using `app.registerExtension`" in Phase B without also warning v2 authors
    - Requires an Axis 2 dispatch mechanism (see below)

---

*Option 2*: `defineExtension({...})` from `@comfyorg/extension-api`

New call site, new package import.

```ts
import { defineExtension } from '@comfyorg/extension-api'

export default defineExtension({
  name: 'MyExt',
  setup(ctx) {
    ctx.onNodeMounted((node: NodeHandle) => { ... })
  }
})
```

- Pros
    - Pure module — testable without running app
    - Clean deprecation: `app.registerExtension` = v1 only; `defineExtension` = v2. No ambiguity, no Axis 2 needed
    - Typed, versioned public surface via npm package
- Cons
    - Authors must change call site AND implementation — double migration burden
    - Package doesn't ship until PKG2–PKG6; no adoption until then
    - `export default` is foreign to all existing extensions
    - `startExtensionSystem()` not wired yet (gap found in MIG1.E4)

---

*Option 3* (align with how core ComfyUI does it): Versioned npm subpath exports (adapted from `feat/import-based-api-versioning`)

New call site, new package import, but import path itself declares version. The researched branch used internal Vite aliases (`@/scripts/app/v1_2`) — impossible for external authors. Viable equivalent: npm subpath exports.

```ts
// v1 compat shim — same behavior as app.registerExtension
import { registerExtension } from '@comfyorg/extension-api/v1'
registerExtension({ name: 'MyExt', nodeCreated(node: LGraphNode) { ... } })

// v2 — pure module
import { registerExtension } from '@comfyorg/extension-api/v2'
registerExtension({ name: 'MyExt', nodeCreated(node: NodeHandle) { ... } })
```

`package.json` `exports` field routes `/v1` and `/v2` to separate entry modules.

- Pros
    - Import path IS the version declaration — no field, no duck-type, no Axis 2
    - v1 and v2 types fully isolated at TS level (separate entry files, no union overloads)
    - Cleanest deprecation: sunset the entire `/v1` subpath
    - Telemetry trivial: which subpath imported = which version
- Cons
    - Same package blocker as CS-B (PKG2–PKG6)
    - Authors must add the import even for v1 compat — higher friction than CS-A during transition
    - `/v1` subpath still wraps `app.registerExtension`; `window.app` dependent underneath
    - Two subpath entry points to maintain

---

## Axis 2: Dispatch mechanism (only relevant for CS-A)

When the call site is `app.registerExtension`, the host needs to know which version an extension declared. Four options:

*Option 1: Explicit version field*

```ts
app.registerExtension({ name: 'MyExt', apiVersion: 'v2', nodeCreated(node: NodeHandle) { ... } })
```

Author adds `apiVersion: 'v2'`. Absent = v1. Host routes entire extension object to v1 or v2 service at registration time.

- Pros: explicit, greppable, mirrors `getExtensionVersionReport` pattern from issue #4668
- Cons: `apiVersion` feels like a leaky internal concern surfaced to authors; all-or-nothing per registration (can't mix v1/v2 hooks in one object)

---

*Option 2: Key name suffix*

```ts
app.registerExtension({
  name: 'MyExt',
  nodeCreated(node: LGraphNode) { ... },    // v1 — still runs
  nodeCreatedV2(node: NodeHandle) { ... }   // v2 — colocated
})
```

No version field. Host calls whichever key is present. Authors migrate one hook at a time.

- Pros: per-hook migration — author can ship `nodeCreatedV2` while `nodeCreated` still runs; no version field; no union types
- Cons: `V2` suffix pollutes hook namespace permanently; deprecating v1 means deprecating a named set of keys, not a version path; doesn't scale to v3

---

*Option 3: Argument type (TS overload + runtime duck-type)*

```ts
// v1 overload
app.registerExtension({ name: 'MyExt', nodeCreated(node: LGraphNode) { ... } })

// v2 overload — same key, different parameter type
app.registerExtension({ name: 'MyExt', nodeCreated(node: NodeHandle) { ... } })
```

Two TS overloads discriminated by whether `nodeCreated`'s parameter is `LGraphNode` or `NodeHandle`. At runtime: host calls with `NodeHandle` for v2-tagged extensions, `LGraphNode` for v1. Requires `apiVersion` tag internally or duck-typing `'entityId' in node` to decide which to pass.

- Pros: zero new fields or keys visible to authors; `LGraphNode` and `NodeHandle` don't overlap so TS overloads are clean; clean deprecation of the `LGraphNode` overload
- Cons: still needs an internal tag (collapses into DM-1 at the implementation level); IDEs sometimes struggle to narrow two overloads; migration guide must explain the type difference clearly

---

*Option 4*: No mechanism - duck-type at hook call time, author writes guard

```ts
app.registerExtension({
  name: 'MyExt',
  nodeCreated(node: LGraphNode | NodeHandle) {
    if ('entityId' in node) { /* v2 */ } else { /* v1 */ }
  }
})
```

Host always calls `nodeCreated` with whatever the current runtime provides. Authors write a type guard inside the hook body.

- Pros: zero object-shape change; no host changes needed; author controls migration timeline entirely
- Cons: once v2 is live the host must pass `NodeHandle` to everyone — forces all authors to update their hook body even if they don't want v2 behavior; `LGraphNode | NodeHandle` union bleeds into every hook body; duck-typing is fragile; no host-side deprecation lever

---

## Fatal collision: DM by `setup` key shape

A natural-seeming variant of DM-2/DM-3: detect v2 by presence of a `setup(ctx)` key. **Not viable.** v1 `ComfyExtension` already has `setup?: (app: ComfyApp) => void`. Both v1 and v2 look like `{ name, setup }` at runtime. No reliable discriminator without argument count checks (fragile) or adding a version field (= DM-1).

## Option space

Combining the axes:

| Option       | Call site                  | Dispatch mechanism                          |
| ------------ | -------------------------- | ------------------------------------------- |
| CS-A + DM-1  | `app.registerExtension`    | `apiVersion: 'v2'` field                    |
| CS-A + DM-2  | `app.registerExtension`    | `nodeCreatedV2` key suffix                  |
| CS-A + DM-3  | `app.registerExtension`    | TS overload on arg type (DM-1 internally)   |
| CS-A + DM-4  | `app.registerExtension`    | Author-side type guard; no host mechanism   |
| CS-B         | `defineExtension(...)`     | N/A — call site is the discriminator        |
| CS-C         | versioned subpath import   | N/A — import path is the discriminator      |

## Recommendation

**CS-A + DM-3 for Phase A; CS-B or CS-C as the long-term preferred form.**

DM-3 surfaces the cleanest author-facing shape (no new fields, no suffix keys) while giving the host a real deprecation lever (the `LGraphNode` overload). Internally it needs a tag (DM-1 at the implementation level), but that's hidden from authors.

1. `app.registerExtension({ nodeCreated(node: NodeHandle) {...} })` works in Phase A. Internal `apiVersion` tag drives dispatch. No package import required.
2. `import { defineExtension } from '@comfyorg/extension-api'` ships in PKG2, documented as recommended for new extensions.
3. When PKG2 ships, evaluate CS-C (versioned subpaths) vs CS-B — import-path-as-version is strictly cleaner than `apiVersion` field if the package is already required.
4. Phase B deprecation warning: "your `nodeCreated` is receiving a `LGraphNode`; update to `NodeHandle` to migrate."
5. Phase D removal: drop the `LGraphNode` overload. `app.registerExtension` with `NodeHandle` keeps working.

**Why not DM-1 (explicit field) for authors:** `apiVersion: 'v2'` is a leaky internal concern. DM-3 hides it.

**Why not DM-2 (suffix keys):** `nodeCreatedV2` pollutes the namespace and doesn't scale. The per-hook migration story is appealing but the cost is permanent.

**Why not DM-4:** host loses the ability to pass `LGraphNode` to anyone once v2 is live — unilateral breaking change for all authors.

**Why not CS-B/CS-C outright:** package doesn't exist yet. Requiring it blocks the behavioral migration. CS-A + DM-3 unblocks I-COORD.1 now.

## Open questions (require user input before decision is accepted)

1. **DM-3 vs DM-1 for authors**: is hiding `apiVersion` as an internal tag acceptable, or should authors explicitly declare it? DM-1 is more honest; DM-3 is cleaner at the call site.

2. **DM-2 per-hook migration appeal**: the ability to migrate one hook at a time is genuinely useful for large extensions. Is that worth the suffix-key cost, or does the all-or-nothing constraint of DM-3 matter less in practice?

3. **CS-C interest**: when PKG2 ships, should we evaluate versioned subpath exports (`/v1`, `/v2`) as the long-term form? Or is the simpler CS-B surface (`defineExtension`) sufficient?

4. **`export default` convention**: with CS-A as a side effect at import time, there's nothing for the loader to autodiscover. Is that fine for the external custom node case?

5. **Docs framing**: lead with `defineExtension` (CS-B) and note `app.registerExtension` as the compat path, or the reverse?

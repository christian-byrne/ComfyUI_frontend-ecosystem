# D3.4 — Reconciling Events API with ECS Command/Query Pattern

> Clarifies how the event-based extension API relates to the ECS architecture.
> Not a new design — a reframing of D3.3 to show they're the same thing.

---

## The Tension

The ECS architecture (ADR 0008 + World Command API doc) defines:
- **Reads** = queries against the World: `world.getComponent(id, Position)`
- **Writes** = serializable Commands: `dispatch(MoveNodeCommand(id, pos))`

The D3.3 events API proposes:
- **Reads** = `node.getPosition()`
- **Writes** = `widget.setValue(42)`
- **Reactions** = `node.on('positionChanged', fn)`

Are these in conflict? **No — they're the same architecture at different
abstraction levels.** But the question is: should the public API expose
the ECS vocabulary (queries, commands) or hide it behind domain methods?

## Answer: The Domain Handles ARE Queries and Commands

`node.getPosition()` **is** a World query. `widget.setValue(42)` **is** a
command dispatch. The domain handle just provides a typed, discoverable
surface instead of making extension authors construct ComponentKeys and
Command objects.

```
Extension calls          Framework translates to         ECS layer
─────────────            ───────────────────────         ─────────
node.getPosition()    →  world.getComponent(id, Position).pos
widget.getValue()     →  world.getComponent(id, WidgetValue).value
widget.setValue(42)   →  executor.run(SetWidgetValue(id, 42))
node.setTitle('foo')  →  executor.run(SetNodeVisual(id, {title:'foo'}))
node.on('executed')   →  system event subscription
```

The extension API IS the command/query API — it just speaks in domain
terms (position, widget value, title) instead of ECS terms (ComponentKey,
Command, EntityId).

## Why This is the Right Level

### If we exposed raw ECS commands:
```ts
// Extension author writes:
import { SetWidgetValue, Position, WidgetValue } from '@comfyorg/comfyui'

nodeCreated(node) {
  const pos = world.getComponent(node.entityId, Position)
  dispatch(SetWidgetValue({ entityId: widgetId, value: 42 }))
  world.onComponentChanged(node.entityId, Position, (newPos) => { ... })
}
```

Problems:
- Extension author needs to know ComponentKeys (`Position`, `WidgetValue`)
- Extension author constructs Command objects manually
- Extension author works with EntityIds directly
- Huge API surface (every component key, every command type)
- Tightly couples extensions to the internal component schema

### If we expose domain handles (current proposal):
```ts
nodeCreated(node) {
  const pos = node.getPosition()
  widget.setValue(42)
  node.on('positionChanged', (newPos) => { ... })
}
```

Benefits:
- Extension author thinks in domain terms
- Commands happen transparently — undo/redo just works
- Component schema can change without breaking extensions
- Small, stable API surface
- Still backed by the full ECS command/query stack

## The Command Pattern is Still There

Every mutation through the domain handle goes through the command executor:

```ts
// Inside the framework (extension author never sees this)
class NodeHandleImpl implements NodeHandle {
  setPosition(pos: Point) {
    this.executor.run({
      type: 'MoveNode',
      nodeId: this.entityId,
      pos,
      execute(world) {
        LayoutSystem.moveNode(world, this.nodeId, this.pos)
      }
    })
  }

  setTitle(title: string) {
    this.executor.run({
      type: 'SetNodeVisual',
      nodeId: this.entityId,
      title,
      execute(world) {
        const visual = world.getComponent(this.nodeId, NodeVisual)
        world.setComponent(this.nodeId, NodeVisual, { ...visual, title: this.title })
      }
    })
  }
}
```

This means:
- **Undo/redo works** — every `setValue()`, `setPosition()`, etc. creates
  a command in the undo log
- **CRDT sync works** — commands are serializable and can be sent over wire
- **Validation works** — systems can reject commands (returned as error to
  the extension)
- **Transactions work** — batched mutations are atomic

The extension gets all the benefits of the command pattern without knowing
it exists.

## Events ARE Component Change Notifications

`node.on('positionChanged', fn)` is the public name for "subscribe to
changes on this entity's Position component." Internally:

```ts
// Framework implementation
on(event: 'positionChanged', fn: (pos: Point) => void) {
  // This IS a World query subscription
  scope.run(() => {
    watch(
      () => world.getComponent(this.entityId, Position),
      (newPos) => fn(newPos.pos)
    )
  })
}
```

The event names map 1:1 to component changes:

| Event name          | World component watched              |
|---------------------|--------------------------------------|
| `positionChanged`   | `Position`                           |
| `sizeChanged`       | `Dimensions`                         |
| `executed`          | `Execution` (or execution result)    |
| `connected`         | `SlotConnection` / `Connectivity`    |
| `widget.change`     | `WidgetValue`                        |

## Where Raw ECS Access Might Appear Later

If the need arises for extensions that work across entities (not bound to
a single node), we can add query-level APIs without changing the
per-entity handle pattern:

```ts
defineExtension({
  name: 'global-analytics',
  setup() {
    // Query across all entities — this would be the ECS query API
    // But this is a future consideration, not needed for initial API
    const allNodes = queryAll(NodeType, Position)
    onQueryChanged(allNodes, (nodes) => { ... })
  }
})
```

This would be additive. The per-entity handle API (nodeCreated + NodeHandle)
covers the vast majority of use cases and doesn't conflict with adding
cross-entity queries later.

## Summary

The events API and the ECS command/query pattern are **the same system at
different layers**:

```
┌──────────────────────────────────────────────┐
│  Extension API (domain handles)              │
│  node.getPosition()  widget.setValue(42)      │
│  node.on('positionChanged', fn)              │
├──────────────────────────────────────────────┤
│  Command/Query Layer                         │
│  executor.run(MoveNode(...))                 │
│  world.getComponent(id, Position)            │
│  watch(() => world.getComponent(...))        │
├──────────────────────────────────────────────┤
│  ECS World                                   │
│  Components, Systems, Transactions           │
└──────────────────────────────────────────────┘
```

Extensions interact with the top layer. The middle and bottom layers are
internal. This is exactly the relationship described in the World Command
API doc: "External callers never call `world.setComponent()` directly —
they submit commands." The domain handles are a higher-level command
submission interface.

# D3.1 — Design Questions: Resolving the API Surface

> **⚠️ SUPERSEDED — historical context only.**
> D3.1 proposed a two-tier API (domain composables vs raw ECS) and a signals-based
> `node.position // Signal<Point>` surface. Both were rejected by later decisions:
> - **D3.2** superseded the two-tier model with a single signals API.
> - **D3.3** (accepted) superseded D3.2's signals model entirely: signals are not
>   in the public API; the surface is event + getter/setter (`node.getPosition()`,
>   `node.on('positionChanged', fn)`). See D3.3.
> - **D6** (accepted) sets the package name as `@comfyorg/extension-api`, not
>   `@comfyorg/comfyui` as proposed here.
> - Open questions in §Q1–Q9 are all resolved in D3.3–D10.
>
> Read this document to understand the reasoning path, not the current design.

> Responses to architectural questions raised during review of D1-D3.

---

## Q1. The API feels "internal" — how do we make it feel natural?

### The problem

The D3 proposal uses terms like `ctx`, `useComponent`, `EffectScope`, `expose` — 
these are implementation concepts. Vue's public API hides all of that. A Vue user 
writes `ref(0)` and `onMounted(() => {...})`, never thinking about `activeSub`, 
`Link` objects, or `EffectScope`. The magic is invisible.

### The fix: Two API tiers

**Tier 1 — "Just works" API** (what 90% of extensions use):

```ts
// The extension author writes this. No ctx, no scope, no effect.
defineNodeExtension('my-extension', {
  // This function IS the setup function. It runs once per node instance.
  // Everything declared here is hoisted — lives as long as the node.
  nodeCreated(node) {
    // Plain reactive values — identical to Vue's ref/computed
    const position = useNodePosition(node)        // Ref<Point>
    const widgetValues = useWidgetValues(node)     // Ref<Record<string, any>>
    const isSelected = useNodeSelected(node)       // Ref<boolean>
    
    // Familiar watch — same API as Vue
    watch(position, (pos) => {
      console.log('node moved to', pos)
    })
    
    // Lifecycle — same pattern as Vue
    onNodeRemoved(() => {
      console.log('goodbye')
    })
  }
})
```

No `ctx` argument. No `useComponent(entityId, ComponentKey)`. No branded IDs. 
The framework provides **domain-specific composables** (`useNodePosition`, 
`useWidgetValues`, `useNodeSelected`) that wrap the ECS internals.

This is exactly what Vue does: `useRoute()` hides router internals, 
`useStore()` hides Pinia internals. The user never sees the raw machinery.

**Tier 2 — "Power user" API** (for extensions that need full ECS access):

```ts
defineNodeExtension('advanced-extension', {
  nodeCreated(node) {
    // Tier 2: direct ECS access for power users
    const world = useWorld()
    const connectivity = useComponent(node.entityId, Connectivity)
    const customQuery = useQuery(NodeType, Position)
    
    // Submit mutations as commands
    dispatch(MoveNodeCommand(node.entityId, [100, 200]))
  }
})
```

The key insight: **the `node` argument IS the context**, but it's a 
domain-shaped object, not a raw context bag. Internally it carries the entityId 
and scope, but the extension author sees "a node."

### How does Vue do this?

Vue users write `const count = ref(0)` — they don't write 
`const count = ctx.createRef(0)`. The functions are **free-standing** and use the 
implicit `currentInstance` to bind to the right component.

We should do the same:

```ts
// These are free-standing functions, not methods on a context object
function useNodePosition(node: NodeHandle): Readonly<Ref<Point>> { ... }
function useWidgetValues(node: NodeHandle): Readonly<Ref<Record<string, any>>> { ... }
function watch(source, cb, options?) { ... }  // Re-export Vue's watch
function onNodeRemoved(fn: () => void) { ... }
```

Internally, `useNodePosition` does:
1. Gets `currentExtensionScope` (the implicit global, like Vue's `currentInstance`)
2. Creates a ref bound to `world.getComponent(node.entityId, Position)`
3. Registers it in the scope for auto-cleanup
4. Returns a readonly ref

The extension author never sees any of this.

## Q2. Does currentInstance / hoisted state conflict with ECS direction?

### Short answer: No conflict, but a micro-adjustment is needed.

The ECS direction says: "entities are IDs, components are data, systems are 
behavior." The Vue-style implicit context pattern (`currentInstance`) is about 
**how the framework wires things up**, not about where data lives.

### What's compatible

- **Data lives in the World** (ECS principle) ✓
- **Behavior lives in systems** (ECS principle) ✓ — extension hooks ARE a kind 
  of system
- **Implicit context for setup-time registration** (Vue pattern) — this is just 
  a wiring mechanism, orthogonal to where data is stored
- **EffectScope per extension+entity** — this is lifecycle management, not data 
  storage

### The micro-adjustment

In Vue, `currentInstance` maps to a **component instance** which has its own 
local state (`setupState`). In our system, we need to be clear that the 
extension's "hoisted state" is:

1. **Reactive references TO World data** — not copies, not local state
2. **Extension-local derived state** — computeds, local refs for UI state that 
   doesn't belong in the World
3. **Registered callbacks** — lifecycle hooks, watchers

The rule: **World is the source of truth. Extensions observe and command it, 
they don't own entity data.** Extension-local state (like "is my panel open") 
is fine as local refs. Entity state (position, widget values) always lives in 
the World.

This is actually a cleaner separation than Vue has. In Vue, components own their 
own state AND can inject external state. In our system, entities own state (via 
World), extensions just have reactive views into it.

## Q3. KeepAlive persistent state vs. serialization boundaries

### The distinction

**Vue KeepAlive**: The component's reactive state (refs, computeds, watchers) 
survives DOM unmount/remount. The component instance stays in memory. When 
re-mounted, it resumes with the same state.

**ComfyUI serialization boundaries**: When a node moves between graph scopes 
(e.g., packed into a subgraph), the current system destroys and recreates it. 
State is preserved via serialization — `serialize()` → JSON → `configure()`. 
Any in-memory state not captured by serialization is lost.

### The reconciliation

Under ECS, these converge:

1. **Entity IDs are stable across scope changes.** When a node is packed into a 
   subgraph, its `NodeEntityId` doesn't change — only its `graphScope` tag 
   changes. This is explicitly stated in the subgraph-boundaries doc: "Entities 
   keep their IDs throughout. Only the graphScope tag changes."

2. **Extension scopes are keyed by entity ID.** The scope for 
   `extension:my-ext + entity:Node42` persists as long as `Node42` exists in 
   the World — regardless of which `graphScope` it's in.

3. **No serialize/deserialize cycle needed.** Moving a node between scopes is a 
   metadata change (`graphScope: root` → `graphScope: subgraph-1`), not a 
   destroy+recreate. So the extension's reactive refs, watchers, and derived 
   state naturally survive the transition.

4. **Widget promotion (Candidate A)** makes this even cleaner: promoting a 
   widget adds a typed input to the subgraph interface. The widget entity keeps 
   its ID. The extension scope keeps watching the same `WidgetEntityId`. No 
   proxy objects, no reconciliation.

**Where serialization still matters**: When a workflow is saved to disk and 
reloaded, extension scopes ARE recreated. But this is the same as Vue — closing 
and reopening a page destroys all component instances. The extension's `setup` 
function runs again on reload, re-establishing reactive bindings.

### The practical impact

```
BEFORE (current):
  Node in graph → pack into subgraph → serialize → deserialize → recreate
  Extension state: LOST (unless extension manually serializes)

AFTER (ECS + hoisted state):
  Node in graph → change graphScope tag → node still in World
  Extension scope: PRESERVED (same entityId, same scope)
  
  Node promoted widget → add interface input → widget still in World
  Extension watching that widget: PRESERVED
```

## Q4. Who accesses `expose()`d stuff and how?

### Re-evaluation: We probably don't need `expose()`.

In Vue, `expose()` is for parent components to access child component methods 
via template refs. Our system doesn't have this parent-child relationship 
between extensions.

**What we actually need instead:**

1. **Extension-to-extension communication**: Use `provide/inject` (or the ECS 
   equivalent). Extension A provides data, Extension B injects it. This is how 
   Vue plugins share state.

2. **Extension state visible to the system**: The return value of the setup 
   function. If an extension returns `{ customProperty: ref('hello') }`, the 
   framework can make that accessible to other extensions or to debugging tools.

3. **Custom widget state accessible to the framework**: This happens naturally 
   through the World. Widget extensions write to `WidgetValue` components via 
   commands. Other systems read `WidgetValue`.

**Recommendation**: Drop `expose()` from the initial API. If inter-extension 
communication is needed, use `provide(key, value)` at the extension level and 
`inject(key)` from other extensions. This matches Vue's pattern exactly.

## Q5. If extensions can't import Vue reactivity, what do we do?

### The constraint

Custom node extensions load as external JavaScript. They can't `import { watch, computed } from 'vue'`. They may not even know Vue exists. The `onAuth*` 
hooks are the prior art — they're plain callbacks, no reactivity knowledge 
needed.

### Three tiers of solution

**Tier A — Plain callbacks (simplest, covers most cases):**

```ts
defineNodeExtension('simple-ext', {
  nodeCreated(node) {
    // Callback-based API — no reactivity imports needed
    // These are provided by the framework, not by Vue
    node.onPositionChanged((pos) => { ... })
    node.onWidgetValueChanged('seed', (value) => { ... })
    node.onRemoved(() => { ... })
  }
})
```

Internally, `node.onPositionChanged` creates a Vue `watch` on the Position 
component. But the extension author never sees `watch` or `Ref`. It's just a 
callback registration.

This is equivalent to how `addEventListener` wraps the DOM's event system. 
Same pattern as `onAuthUserResolved`.

**Tier B — Signals API (framework-provided, no Vue import needed):**

```ts
defineNodeExtension('signal-ext', {
  nodeCreated(node) {
    // Framework provides signal-like API, not Vue's ref/watch
    const pos = node.position       // ReadonlySignal<Point>
    const seed = node.widget('seed') // ReadonlySignal<number>
    
    // effect() provided by framework, not Vue
    effect(() => {
      console.log('position is', pos.value, 'seed is', seed.value)
    })
  }
})
```

The framework provides its own `effect()` that wraps Vue's `watchEffect`. The 
`Signal` type is a simplified read-only wrapper around Vue's `Ref`. Extensions 
don't import from Vue — they use framework-provided primitives.

**Tier C — Full Vue reactivity (opt-in for Vue-aware extensions):**

```ts
// For extensions that ARE Vue-aware (e.g., our own core extensions)
import { watch, computed, ref } from 'vue'

defineNodeExtension('vue-ext', {
  nodeCreated(node) {
    const pos = useNodePosition(node)  // Returns a real Vue Ref
    const doubled = computed(() => pos.value[0] * 2)
    watch(pos, (p) => { ... })
  }
})
```

### Recommendation

Start with **Tier A** (callbacks) as the primary public API. It requires zero 
framework knowledge and matches the existing `onAuth*` pattern. Add **Tier B** 
(signals) for extensions that want reactive derived state. Leave **Tier C** 
available for internal/advanced use.

The nice thing: all three tiers work simultaneously because they're all backed 
by the same Vue reactivity underneath. An extension can mix callbacks and 
signals freely.

## Q6. What exactly do the D3 architecture steps mean?

> "1. Create/reuse EffectScope for this extension+entity"
> "2. Set currentExtension = this scope"
> "3. Build HookSetupContext with entity-bound APIs"

### Step-by-step, concretely

Let's say extension "color-picker" registered `nodeCreated`. A new KSampler 
node (entityId = Node42) is added to the graph.

**Step 1: Create/reuse EffectScope**

```ts
// The framework does this (inside invokeExtensions):
const scopeKey = 'color-picker:Node:42'

let scope = scopeRegistry.get(scopeKey)
if (!scope) {
  scope = new EffectScope(true)  // Vue's EffectScope, detached
  scopeRegistry.set(scopeKey, scope)
  
  // When Node42 is deleted from the World, auto-cleanup:
  onEntityRemoved(Node42, () => {
    scope.stop()  // Stops all watchers, computed, runs cleanups
    scopeRegistry.delete(scopeKey)
  })
}
```

An `EffectScope` is Vue's container for reactive effects. Any `watch()`, 
`computed()`, or `watchEffect()` created while this scope is active gets 
collected into it. When the scope stops, all those effects are automatically 
cleaned up.

"Create/reuse" means: first time `nodeCreated` fires for this extension+node 
pair, create a new scope. If somehow the same hook fires again for the same 
pair (e.g., during hot reload), reuse the existing scope.

**Step 2: Set currentExtension = this scope**

```ts
// The framework does this:
const previousScope = currentExtensionScope
currentExtensionScope = scope
scope.on()  // Makes this the active EffectScope for Vue's system
```

This is the equivalent of Vue's `setCurrentInstance(instance)`. It sets a 
module-level global so that any `watch()`, `onRemoved()`, or `effect()` 
called during the extension's hook can automatically register in the right 
scope — without the extension having to pass the scope around.

"Set currentExtension" literally means: set a global variable to point to 
this scope, so that the free-standing API functions know which scope to 
register effects in.

**Step 3: Build node handle with entity-bound APIs**

```ts
// The framework does this:
const node: NodeHandle = {
  entityId: Node42,
  
  // Pre-bound to this specific entity:
  get position() { return useNodePosition(Node42) },
  
  // Callback-style API (Tier A):
  onPositionChanged(cb) { 
    watch(useComponent(Node42, Position), cb) 
  },
  onRemoved(cb) { 
    onScopeDispose(cb)  // runs when scope stops
  },
  
  // Widget access:
  widget(name) { return useWidgetValue(Node42, name) },
  
  // Power-user escape hatch:
  entityId: Node42,
}
```

"Entity-bound" means: the APIs on this object are pre-bound to Node42. The 
extension doesn't need to pass entity IDs around — `node.position` already 
knows which node it's reading from.

**Then the extension's hook is called:**

```ts
scope.run(() => {
  extension.nodeCreated(node)
  // Inside here, any watch/computed/effect auto-registers in scope
})

// Restore previous scope
currentExtensionScope = previousScope
```

## Q7. Should we abstract away `ctx.useComponent` semantics?

### Yes — for Tier 1 (most extensions).

`useComponent(entityId, ComponentKey)` is a power-user API (Tier 2). Most 
extensions should never see it.

Instead, the `node` handle provides domain-specific properties:

```ts
// What most extensions see:
node.position         // Ref<Point>  (wraps useComponent(id, Position))
node.title            // Ref<string> (wraps useComponent(id, NodeVisual).title)
node.widgets          // Record<string, Ref<any>> (wraps WidgetValue queries)
node.type             // string (from NodeType component)
node.isSelected       // Ref<boolean>

// What power users can access:
const world = useWorld()
const raw = useComponent(node.entityId, Position)  // Direct ECS access
```

This is exactly the pattern Vue uses: `props.foo` is simpler than 
`instance.vnode.props.foo`. The framework provides convenient shapes; the raw 
internals are available when needed.

## Q8. What about when setup runs before the node is active on the graph?

### This maps to Vue's `setup()` vs `onMounted()` distinction.

In Vue:
- `setup()` runs immediately — the component exists in memory but isn't in the 
  DOM yet
- `onMounted()` runs after the component enters the DOM

In our system:
- `nodeCreated()` runs when the entity is created in the World — it exists as 
  data but may not be rendered yet
- `onNodeMounted()` would run after the node is rendered on the canvas (first 
  paint)
- `onNodeActive()` would run when the node enters the active/visible graph scope

**The entity IS in the World when `nodeCreated` fires.** All component data 
exists. Reactive refs work. You can read position, widget values, etc. What 
doesn't exist yet: the DOM elements for the node's canvas representation.

For most extensions, this doesn't matter — they're reacting to data, not DOM. 
For extensions that need DOM access (e.g., custom widget rendering), they would 
use `onNodeMounted()` which fires after the render system processes the entity.

```ts
nodeCreated(node) {
  // Entity exists in World, data is available
  const pos = node.position  // Works — Position component exists
  
  // But if you need DOM elements:
  onNodeMounted(() => {
    // Now the canvas has rendered this node
    // DOM-related setup goes here
  })
}
```

## Q9. Widgets, mirroring, subgraph promotion — how does it fit?

### This is where the ECS model really shines.

Under the ECS + hook system, widgets have their own entity lifecycle:

```ts
defineWidgetExtension('my-slider', {
  // This is the "SFC-like" registration Christian mentioned
  // The widget type is registered once, instances are per-node
  
  widgetCreated(widget) {
    // widget.entityId is a WidgetEntityId
    // widget.parentNode is a NodeEntityId
    const value = widget.value          // Ref<number>
    const options = widget.options      // Ref<WidgetOptions>
    
    onWidgetRemoved(() => { ... })
    
    // Return a render function (SFC-like)
    return {
      render(container: HTMLElement) {
        // Mount Vue component or vanilla DOM
        // The container is managed by the framework
      }
    }
  }
})
```

### Subgraph promotion (Candidate A — connections-only)

When a widget is "promoted" to the subgraph parent:

1. A typed input is added to `SubgraphStructure.interface`
2. The SubgraphNode gets a NEW widget entity (via type→widget mapping)
3. The INTERIOR widget's input is replaced by a boundary connection

**From the extension's perspective:**

```
Interior node's seed widget (WidgetEntity #55):
  - widgetCreated fires → extension sets up reactive bindings
  - Widget is connected to boundary → value now comes from parent
  - Extension's watch on widget.value still fires (value changed source)
  - Extension scope: ALIVE (entity still in World)

SubgraphNode's new seed widget (WidgetEntity #78):
  - widgetCreated fires → SAME extension type fires for new instance
  - New scope created for extension:'my-slider' + entity:Widget78
  - Independent reactive state

User changes value on SubgraphNode's widget (#78):
  - Value flows through boundary link to interior node input
  - Interior widget sees updated value through the connection
```

**The key: each widget instance gets its own extension scope.** When a widget 
is mirrored/promoted, it's a new entity with a new scope. The extension code 
runs exactly the same way — it doesn't know or care that it's a "promoted" 
widget vs. an "original" widget. The ECS handles the value flow through 
boundary connections.

### App mode (widget without a node)

Under the ECS model, a widget entity can exist with a `parentNode` that points 
to a node in app mode (which is just a node with a different view):

```ts
defineWidgetExtension('my-slider', {
  widgetCreated(widget) {
    const value = widget.value
    
    // This works identically in graph mode and app mode:
    watch(value, (v) => { ... })
    
    return {
      render(container) {
        // Same render function, different container
        // In graph mode: canvas widget slot
        // In app mode: app layout slot
      }
    }
  }
})
```

The extension doesn't need two code paths. The entity is the same. The scope is 
the same. Only the render target changes — and that's the framework's 
responsibility, not the extension's.

### DOM moves (graph↔app, subgraph enter/exit)

When the DOM container for a widget changes (because the user switched between 
graph view and app view):

1. The framework detaches the widget's rendered DOM from the old container
2. Attaches it to the new container
3. **Nothing else changes** — the entity is the same, the scope is alive, all 
   watchers are still running

This is exactly Vue's `<Teleport>` or `<KeepAlive>` behavior. The reactive 
state is decoupled from the DOM location.

---

## Summary of Recommendations

1. **Two-tier API**: Domain composables (Tier 1) over raw ECS access (Tier 2)
2. **No explicit ctx**: Use free-standing functions + implicit scope (Vue pattern)
3. **Callbacks as primary API**: `node.onPositionChanged(cb)` — no Vue imports needed
4. **Drop `expose()`**: Use `provide/inject` for inter-extension communication
5. **Entity IDs are stable**: Scope survives graph scope changes (no serialize/deserialize)
6. **Widget extensions are SFC-like**: One registration, per-instance scopes, framework manages containers
7. **Promotion = new entity**: Same extension code runs for promoted and original widgets

# D3.3 — Events Over Signals

> Supersedes D3.2's signal-based API. Decision: pure event/callback API.

---

## The Honest Assessment

Looking at every real extension pattern — both our core extensions and community 
ones — the usage is always the same shape:

```ts
// 1. An event fires (node created, executed, widget changed)
// 2. Extension reads some current state
// 3. Extension performs a side effect (set a value, toggle visibility, etc.)
```

No extension composes multiple reactive sources. No extension derives computed 
values from other reactive values. No extension needs auto-tracking. They 
respond to discrete events and do things.

**Signals solve a problem extensions don't have.** They add conceptual overhead 
(`.value`, auto-tracking, implicit subscription) for zero practical benefit in 
this context. Events are universally understood, trivially typed, and match 
exactly what extensions actually do.

## Evidence: Real Extension Patterns

### Pattern 1: React to node creation, set initial state
```ts
// ImageCrop (current)
nodeCreated(node) {
  if (node.constructor.comfyClass !== 'ImageCropV2') return
  node.hideOutputImages = true
  node.setSize([Math.max(oldWidth, 300), Math.max(oldHeight, 450)])
}
```
This is: event fires → read state → write state. No reactivity needed.

### Pattern 2: React to execution, update widget value
```ts
// PreviewAny (current)
nodeType.prototype.onExecuted = function(message) {
  for (const previewWidget of previewWidgets) {
    previewWidget.value = message.text ?? ''
  }
}
```
This is: event fires → set value. No reactivity needed.

### Pattern 3: React to widget change, toggle other widgets
```ts
// PreviewAny (current)
showAsPlaintextWidget.widget.callback = (value) => {
  showValueWidget.hidden = !value
  showValueWidgetPlain.hidden = value
}
```
This is: event fires → set state on other widgets. No reactivity needed.

### Pattern 4: Override serialization behavior
```ts
// DynamicPrompts (current)
widget.serializeValue = (workflowNode, widgetIndex) => {
  return processDynamicPrompt(widget.value)
}
```
This is: register a callback for a specific lifecycle moment. No reactivity needed.

## The API: Events + Getters/Setters

```ts
import { defineNodeExtension } from '@comfyorg/comfyui'

defineNodeExtension({
  name: 'Comfy.PreviewAny',
  nodeTypes: ['PreviewAny'],

  nodeCreated(node) {
    // Read current state — plain getters, no .value
    const pos = node.getPosition()      // Point
    const size = node.getSize()         // Size

    // Add widgets — returns a WidgetHandle
    const preview = node.addWidget('MARKDOWN', 'preview_markdown', '')
    const previewPlain = node.addWidget('STRING', 'preview_text', '', {
      multiline: true
    })
    const modeToggle = node.addWidget('BOOLEAN', 'previewMode', false, {
      labelOn: 'Markdown',
      labelOff: 'Plaintext'
    })

    preview.setHidden(true)

    // React to events — plain callbacks
    modeToggle.on('change', (value) => {
      preview.setHidden(!value)
      previewPlain.setHidden(value)
    })

    node.on('executed', (output) => {
      const text = output.text ?? ''
      const content = Array.isArray(text) ? text.join('\n\n') : text
      preview.setValue(content)
      previewPlain.setValue(content)
    })

    node.on('removed', () => {
      // Explicit cleanup if needed (framework also auto-cleans)
    })
  }
})
```

### NodeHandle (event-based)

```ts
interface NodeHandle {
  // Identity
  readonly entityId: NodeEntityId
  readonly type: string
  readonly comfyClass: string

  // Read current state (plain getters)
  getPosition(): Point
  getSize(): Size
  getTitle(): string
  getMode(): NodeMode
  getProperties(): Record<string, unknown>
  isSelected(): boolean

  // Write state (via commands internally)
  setPosition(pos: Point): void
  setSize(size: Size): void
  setTitle(title: string): void
  setProperty(key: string, value: unknown): void

  // Widget access
  widget(name: string): WidgetHandle | undefined
  widgets(): WidgetHandle[]
  addWidget(type: string, name: string, defaultValue: any, options?: WidgetOptions): WidgetHandle

  // Events
  on(event: 'removed', fn: () => void): void
  on(event: 'executed', fn: (output: Record<string, any>) => void): void
  on(event: 'connected', fn: (input: SlotInfo, output: SlotInfo) => void): void
  on(event: 'disconnected', fn: (input: SlotInfo, output: SlotInfo) => void): void
  on(event: 'configured', fn: () => void): void
  on(event: 'positionChanged', fn: (pos: Point) => void): void
  on(event: 'sizeChanged', fn: (size: Size) => void): void
}
```

### WidgetHandle (with modelValue-like pattern)

```ts
interface WidgetHandle {
  readonly entityId: WidgetEntityId
  readonly name: string
  readonly widgetType: string

  // modelValue equivalent — the single read/write point
  // All views (node, properties panel, promoted copy) share this
  getValue(): any
  setValue(value: any): void

  // Events
  on(event: 'change', fn: (value: any, oldValue: any) => void): void
  on(event: 'removed', fn: () => void): void

  // Visibility/options
  setHidden(hidden: boolean): void
  isHidden(): boolean
  getOptions(): WidgetOptions
  setOptions(opts: Partial<WidgetOptions>): void

  // Serialization override
  setSerializeValue(fn: (workflowNode: any, widgetIndex: number) => any): void
}
```

### Why `getValue()/setValue()` instead of a signal's `.value`

`widget.getValue()` / `widget.setValue(42)` is the modelValue equivalent. It 
reads from and writes to the World's `WidgetValue` component. All views 
(node widget, properties panel, promoted copy) resolve to the same underlying 
component, so changes from any source fire the `change` event on all handles.

```
User edits on node      → setValue(42) → World updates → 'change' fires on all
User edits in props panel → setValue(42) → World updates → 'change' fires on all
API sets value           → setValue(42) → World updates → 'change' fires on all
```

No `.value` property to learn. No WritableSignal concept. Just get/set + event.

## How This Stays Backed by Vue Reactivity

Internally, the framework still uses Vue's reactivity:

```ts
// Framework internal — extension author never sees this
function createNodeHandle(entityId: NodeEntityId, scope: EffectScope): NodeHandle {
  const listeners = new Map<string, Set<Function>>()

  // Bind Vue watch to event dispatch
  scope.run(() => {
    watch(
      () => world.getComponent(entityId, Position),
      (newPos, oldPos) => {
        listeners.get('positionChanged')?.forEach(fn => fn(newPos.pos))
      }
    )
    // ... same for other watchable properties
  })

  return {
    entityId,
    getPosition: () => world.getComponent(entityId, Position).pos,
    on(event, fn) {
      if (!listeners.has(event)) listeners.set(event, new Set())
      listeners.get(event)!.add(fn)
      // Auto-cleanup when scope stops
      onScopeDispose(() => listeners.get(event)?.delete(fn))
    },
    // ...
  }
}
```

The reactive adapter converts World component changes into events. Extensions 
get the simplicity of callbacks. The framework gets the performance of 
Vue's batched reactivity. Auto-cleanup via EffectScope still works — when the 
node is removed, the scope stops, all watches stop, all listener sets are 
cleared.

## What About the Rare Case That Needs Derived State?

If an extension genuinely needs to combine multiple values (rare based on 
evidence), they just do it in JavaScript:

```ts
nodeCreated(node) {
  let lastPos: Point = node.getPosition()
  let lastSize: Size = node.getSize()

  function checkIfLarge() {
    return lastSize[0] > 300 && lastSize[1] > 200
  }

  node.on('positionChanged', (pos) => { lastPos = pos })
  node.on('sizeChanged', (size) => {
    lastSize = size
    console.log('isLarge:', checkIfLarge())
  })
}
```

Is this less elegant than `derived(() => ...)`? Yes. Does any extension 
actually need this? Based on every pattern I've seen — no.

## What We Explicitly Deprecate

### 1. `beforeRegisterNodeDef`

**Deprecated.** Use `nodeCreated` with `nodeTypes` filter instead.

Every current usage of `beforeRegisterNodeDef` does one of:
- Guard with `if (nodeData.name === 'Foo')` then patch → use `nodeTypes: ['Foo']`
- Patch ALL prototypes → use `nodeCreated` without filter

The new API is per-instance, not per-class. No prototype chain fragility.

### 2. `useChainCallback`

**Deprecated.** The new API uses `node.on(event, fn)` which naturally 
supports multiple listeners. No chaining, no capturing original functions, 
no `apply(this, args)`.

### 3. Direct prototype/instance access

**Deprecated.** `node.pos`, `node.widgets[0].value`, `widget.callback` 
assignments are replaced by `node.getPosition()`, `widget.getValue()`, 
`widget.on('change', fn)`.

### 4. Docs that suggest monkey-patching

All documentation guiding extension authors toward prototype patching or raw 
instance access should be replaced with the new API patterns.

## Migration Path

```
Phase 1: Ship new API alongside old
  - defineNodeExtension + NodeHandle + WidgetHandle available
  - Old ComfyExtension hooks still work unchanged
  - No deprecation warnings yet

Phase 2: Migrate core extensions
  - Convert all src/extensions/core/* to new API
  - This proves the API covers all real use cases
  - Deprecation warnings added to beforeRegisterNodeDef, useChainCallback

Phase 3: Community migration period
  - Documentation updated: new API is primary, old patterns are deprecated
  - Deprecation warnings in console for old patterns
  - Manager badge for "uses stable API"

Phase 4: Remove old patterns
  - beforeRegisterNodeDef removed
  - useChainCallback removed
  - Raw LGraphNode no longer passed to hooks
```

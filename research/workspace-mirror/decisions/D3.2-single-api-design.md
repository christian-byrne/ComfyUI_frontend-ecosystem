# D3.2 — Single API Design: Consolidated Decisions

> **⚠️ SUPERSEDED — historical context only.**
> D3.2 proposed a signals-based public API (`node.position // Signal<Point>`,
> `widget.modelValue // WritableSignal<T>`, `effect()`, `derived()`, `signal()`
> exported from the package). This was rejected in full by **D3.3** (accepted):
> - "Signals solve a problem extensions don't have." The public API is
>   event + getter/setter, not signals. Vue reactivity is the *internal* engine,
>   invisible to extension authors.
> - Package is `@comfyorg/extension-api` (D6), not `@comfyorg/comfyui`.
> - Five open questions at the bottom are all resolved: signal naming → rejected
>   (D3.3); nodeTypes filter → `nodeTypes: string[]` (P1/D3.3); addWidget →
>   `node.addWidget(...)` (P1); provide/inject → deferred (D10a); command
>   vocabulary → domain handles ARE the command API (D3.4).
> - Migration phases here are superseded by D6's 4-phase sunset + D9's strangler fig.
>
> Read this document to understand the reasoning path, not the current design.

> Supersedes the tiered system from D3.1. One API, no tiers.

---

## Core Principle: One API

Every extension author uses the same API surface. No "power user" vs "basic" 
split. No choosing between callbacks vs signals vs raw ECS. One model.

---

## The API Model: Signals + Domain Handles

The API is **signal-based** (not callback-based). Signals are strictly better 
than callbacks because:

- Signals carry the current value AND react to changes (callbacks only fire on 
  change — you need a separate API to read current state)
- Signals compose naturally (derive new signals from existing ones)
- Signals have built-in cleanup via scope
- Signals match the reactive data flow of ECS

Extensions don't import from Vue. The framework provides signal primitives 
as part of its public API.

### What a signal is (for extension authors)

A signal is an object with a `.value` property. Reading `.value` gives the 
current value. The framework re-runs your `effect()` whenever a signal you 
read changes.

```ts
interface Signal<T> {
  readonly value: T
}

interface WritableSignal<T> {
  value: T
}
```

These are backed by Vue refs internally but extension authors never need to 
know that. The framework exports these types as part of the public API.

### Framework-provided primitives

The framework provides three reactive primitives. No Vue import needed:

```ts
// From the ComfyUI public API — NOT from Vue
import { effect, derived, signal } from '@comfyorg/comfyui'

// effect(): run code when signals change
effect(() => {
  console.log('position is', node.position.value)
  // ^ auto-tracks node.position, re-runs when it changes
})

// derived(): create a computed signal from others
const isLeftHalf = derived(() => node.position.value[0] < 500)

// signal(): create local mutable state (for extension-private state only)
const myCounter = signal(0)
myCounter.value++
```

Implementation: `effect` = `watchEffect`, `derived` = `computed`, 
`signal` = `ref`. Thin wrappers, renamed for domain clarity.

## Node Extension API

```ts
import { defineNodeExtension, effect, derived } from '@comfyorg/comfyui'

defineNodeExtension({
  name: 'my-extension',

  // Runs once per node instance. Everything declared here is hoisted.
  nodeCreated(node) {
    // node is a NodeHandle — domain-shaped, pre-bound to this entity.
    // All properties are signals (readonly unless noted).

    node.position        // Signal<Point>
    node.size            // Signal<Size>
    node.title           // Signal<string>
    node.type            // string (static, not reactive)
    node.isSelected      // Signal<boolean>
    node.mode            // Signal<NodeMode>

    // Widget access — each widget value is a WritableSignal
    const seed = node.widget('seed')     // WritableSignal<number> | undefined
    const model = node.widget('model')   // WritableSignal<string> | undefined

    // React to changes
    effect(() => {
      console.log('seed changed to', seed?.value)
    })

    // Derive new values
    const isLarge = derived(() => {
      const [w, h] = node.size.value
      return w > 300 && h > 200
    })

    // Lifecycle
    node.onRemoved(() => {
      // Cleanup — also happens automatically via scope
    })

    // Connections
    node.onConnected((input, output) => { ... })
    node.onDisconnected((input, output) => { ... })

    // Execution results
    node.onExecuted((output) => { ... })
  }
})
```

### What `node` is (NodeHandle)

`NodeHandle` is a domain object the framework creates for each 
extension+node invocation. It's not the raw `LGraphNode` and not a raw 
ECS context. It's a purpose-built API surface.

```ts
interface NodeHandle {
  // Identity (static)
  readonly entityId: NodeEntityId
  readonly type: string
  readonly comfyClass: string

  // Reactive state (signals bound to World components)
  readonly position: Signal<Point>
  readonly size: Signal<Size>
  readonly title: Signal<string>
  readonly isSelected: Signal<boolean>
  readonly mode: Signal<NodeMode>
  readonly properties: Signal<Record<string, unknown>>

  // Widget access
  widget(name: string): WritableSignal<any> | undefined
  widgets(): ReadonlyArray<WidgetHandle>

  // Lifecycle callbacks
  onRemoved(fn: () => void): void
  onConnected(fn: (input: SlotInfo, output: SlotInfo) => void): void
  onDisconnected(fn: (input: SlotInfo, output: SlotInfo) => void): void
  onExecuted(fn: (output: Record<string, any>) => void): void
  onConfigured(fn: () => void): void
}
```

Internally, `node.position` is `readonly(toRef(() => world.getComponent(entityId, Position)))` bound to Vue's dep-tracking. But the extension sees `Signal<Point>`.

## Widget Extension API

Widgets use a `defineModel`-equivalent pattern. The widget's value is a 
`WritableSignal` — the framework handles synchronizing this with the World's 
`WidgetValue` component and any other views (properties panel, mirrored 
widgets).

```ts
import { defineWidgetExtension, effect, derived } from '@comfyorg/comfyui'

defineWidgetExtension({
  name: 'my-slider',
  type: 'MY_SLIDER',

  // Runs once per widget instance
  widgetCreated(widget) {
    // widget.modelValue is the v-model equivalent.
    // WritableSignal — reads from World, writes update World.
    // Properties panel, promoted copies, any mirror all share this source.
    widget.modelValue           // WritableSignal<number>
    widget.options              // Signal<WidgetOptions>
    widget.parentNode           // NodeHandle (or undefined in app mode)

    // React to value changes (from ANY source — user, properties panel, 
    // promoted widget, API)
    effect(() => {
      console.log('slider value:', widget.modelValue.value)
    })

    // Lifecycle
    widget.onRemoved(() => { ... })

    // Return render function
    return {
      render(container: HTMLElement) {
        // Mount DOM. container is managed by framework.
        const input = document.createElement('input')
        input.type = 'range'
        input.value = String(widget.modelValue.value)
        input.oninput = () => {
          widget.modelValue.value = Number(input.value)
        }
        container.appendChild(input)

        // Update DOM when value changes from other sources
        effect(() => {
          input.value = String(widget.modelValue.value)
        })
      }
    }
  }
})
```

### How `modelValue` handles mirroring

`widget.modelValue` is a single `WritableSignal` backed by the World's 
`WidgetValue` component for this `WidgetEntityId`. Every view of this 
widget — on the node, in the properties panel, in a promoted copy — reads 
and writes the same `WidgetValue` component.

```
┌──────────────────────────────────────────────────────────┐
│                   World: WidgetValue #55                  │
│                   { value: 42 }                          │
│                                                          │
│        ┌───────────────┬────────────────┐                │
│        ▼               ▼                ▼                │
│   Node widget    Properties panel  Promoted copy         │
│   modelValue     modelValue        modelValue            │
│   (same signal)  (same signal)     (same signal)         │
│                                                          │
│   Write from any → World updates → all others react      │
└──────────────────────────────────────────────────────────┘
```

No store abstraction to learn. `modelValue.value = 42` from any view 
updates the World. All other views auto-update because they read the same 
signal.

Under promotion Candidate A (connections-only), the promoted widget on the 
SubgraphNode is a NEW widget entity with its own `WidgetValue`. The value 
flows through a boundary connection, not through shared signal identity. 
The extension code is identical — it reads/writes `widget.modelValue` — but 
the plumbing underneath is a link rather than a shared reference.

## What We Explicitly Don't Support

### Prototype monkey-patching via `beforeRegisterNodeDef`

**Current pattern** (will be deprecated):

```ts
// TODAY: Extensions patch the node CLASS before any instances exist
beforeRegisterNodeDef(nodeType, nodeData) {
  const origOnNodeCreated = nodeType.prototype.onNodeCreated
  nodeType.prototype.onNodeCreated = function() {
    origOnNodeCreated?.apply(this)
    // add widgets, modify behavior for ALL instances of this type
  }
}
```

**What this is doing**: Modifying ALL instances of a node type by patching 
the class prototype. The two use cases are:

1. **Adding widgets/behavior to specific node types** (e.g., PreviewAny adds 
   markdown + plaintext widgets to the PreviewAny node type)
2. **Adding behavior to ALL node types** (e.g., WidgetInputs adds 
   `convertWidgetToInput` to every node)

**New API equivalents:**

For (1) — **type-specific behavior**: Use `nodeCreated` with a type filter. 
The `nodeCreated` hook already fires per-instance. Checking `node.type` is 
equivalent to the `if (nodeData.name === 'PreviewAny')` guard that most 
`beforeRegisterNodeDef` implementations use anyway:

```ts
defineNodeExtension({
  name: 'Comfy.PreviewAny',
  // Optional: filter to specific node types (avoids running for every node)
  nodeTypes: ['PreviewAny'],
  
  nodeCreated(node) {
    // Per-instance setup, same as what the prototype patch did
    const preview = node.addWidget('MARKDOWN', 'preview_markdown', '')
    
    node.onExecuted((message) => {
      preview.modelValue.value = message.text ?? ''
    })
  }
})
```

For (2) — **ALL-types behavior**: Same `nodeCreated` without a filter. It 
fires for every node instance:

```ts
defineNodeExtension({
  name: 'Comfy.WidgetInputs',
  // No nodeTypes filter = runs for every node
  
  nodeCreated(node) {
    node.onConfigured(() => {
      // Handle widget input cleanup for this specific instance
    })
  }
})
```

**Why this is fine:**
- `beforeRegisterNodeDef` patches run once per CLASS. `nodeCreated` runs 
  once per INSTANCE. In practice, the prototype patches set up closures over 
  `this` anyway — they're already effectively per-instance.
- The `nodeTypes` filter gives us the optimization of not running for 
  irrelevant node types (prototype patching runs for ALL types even when 
  guarded by an `if` check).
- No more `useChainCallback`, no more capturing `origOnNodeCreated`, no 
  more prototype chain fragility.

**Migration path**: `beforeRegisterNodeDef` continues to work during 
deprecation. Extensions can migrate at their own pace. The framework logs 
deprecation warnings when prototype mutations are detected.

### Direct class instance access

Extensions no longer receive `LGraphNode` instances. They receive 
`NodeHandle` objects. If an extension tries to access `node.pos` (LGraphNode 
property), it gets `undefined` — they should use `node.position.value`.

During migration, the legacy hooks (`nodeCreated(node: LGraphNode, app)`) 
still fire with the raw instance. New hooks (`defineNodeExtension.nodeCreated(node: NodeHandle)`) fire with the handle. Both can coexist.

### `setup()` hook for global initialization

The existing `setup(app)` hook on `ComfyExtension` is retained as-is for 
global extension initialization. It does NOT receive ECS APIs because it 
runs once at app startup, before any entities exist. Extensions that need 
ECS access use `nodeCreated`, `widgetCreated`, etc.

```ts
// STILL WORKS for global setup (API clients, event listeners, etc.)
app.registerExtension({
  name: 'my-ext',
  setup(app) {
    // Global initialization — no ECS, no nodes yet
    api.addEventListener('status', (e) => { ... })
  }
})

// NEW: per-entity reactive hooks
defineNodeExtension({
  name: 'my-ext',
  nodeCreated(node) {
    // Per-node reactive setup with signals
  }
})
```

## Detailed Walkthrough: What the Framework Does

When `defineNodeExtension` is called and a node is created:

### 1. Create EffectScope for this extension+entity

An **EffectScope** is a cleanup container. Everything reactive you create 
inside it (effects, derived signals, callbacks) gets automatically disposed 
when the scope ends.

Think of it like `addEventListener` that auto-removes when the node is 
deleted. You never have to manually unsubscribe.

```ts
// Framework internal — extension author never sees this
const scope = new EffectScope()

// Later, when node is removed from World:
scope.stop()  // All effects/derived/callbacks auto-disposed
```

The scope is keyed by `extensionName + entityId`. One scope per extension 
per node instance.

### 2. Activate the scope

The framework sets this scope as the "current" scope so that any reactive 
primitives (`effect()`, `derived()`) created during the hook automatically 
register in it.

```ts
// Framework internal
scope.run(() => {
  // Inside here, effect() and derived() auto-register in this scope
  extension.nodeCreated(nodeHandle)
})
```

This is why extension authors don't need to pass a scope around or call 
cleanup functions. The framework handles it.

### 3. Build the NodeHandle

The NodeHandle is a plain object with signal properties pre-bound to this 
specific entity in the World:

```ts
// Framework internal
const nodeHandle: NodeHandle = {
  entityId: nodeId,
  type: world.getComponent(nodeId, NodeType).type,
  
  // Each of these is a signal bound to a specific World component
  position: createSignal(() => world.getComponent(nodeId, Position).pos),
  size: createSignal(() => world.getComponent(nodeId, Dimensions).size),
  title: createSignal(() => world.getComponent(nodeId, NodeVisual).title),
  
  // Widget access creates WritableSignals bound to WidgetValue components
  widget(name) {
    const widgetEntity = findWidgetByName(nodeId, name)
    if (!widgetEntity) return undefined
    return createWritableSignal(
      () => world.getComponent(widgetEntity, WidgetValue).value,
      (v) => dispatch(SetWidgetValueCommand(widgetEntity, v))
    )
  },
  
  // Lifecycle callbacks register in the current scope
  onRemoved(fn) { onScopeDispose(fn) },
  onExecuted(fn) { /* register execution result listener */ },
  // ...
}
```

The extension author sees: "I got a `node` with reactive properties." They 
don't see World, EffectScope, Dep, or any framework internals.

## How This Relates to `onAuth*` Prior Art

The `onAuth*` callbacks are the simplest form of our pattern — plain 
callbacks dispatched by iterating registered extensions:

```ts
// Current: imperative dispatch
for (const ext of extensions) {
  ext.onAuthUserResolved?.(user, app)
}
```

The new system generalizes this. Instead of the framework manually iterating 
and calling, extensions declare what they want to react to, and the reactive 
system handles the rest.

But `onAuth*` style callbacks still work. They're just lifecycle callbacks on 
the extension, not per-entity reactive hooks:

```ts
defineNodeExtension({
  name: 'my-ext',
  
  // Global lifecycle (same as current onAuth*)
  onAuthUserResolved(user) { ... },
  
  // Per-entity reactive hooks (new)
  nodeCreated(node) { ... }
})
```

## Open Questions for Next Session

1. **`signal()` naming**: Is `signal/derived/effect` the right vocabulary, or 
   should we use something more ComfyUI-specific? (TC39 Signals proposal uses 
   `Signal.State` / `Signal.Computed` — we could align with that.)

2. **Node type filtering**: Is `nodeTypes: ['PreviewAny']` the right API for 
   type-specific extensions, or should it be a predicate function?

3. **Widget `addWidget` API**: How does an extension dynamically add widgets 
   to a node via the new API? (Currently done via prototype patching in 
   `beforeRegisterNodeDef`.) Proposal: `node.addWidget(type, name, defaultValue)`.

4. **Inter-extension communication**: Do we need `provide/inject` style APIs, 
   or is the World sufficient (extensions communicate by reading/writing 
   shared components)?

5. **Command vocabulary**: What commands ship in the initial public API? 
   (`SetWidgetValue`, `MoveNode`, `ConnectSlots`, ...) These define what 
   extensions can mutate.

# D3.5 — Reactive Dispatch & Scope Alignment with Vue Core

> Resolves two open questions:
> 1. Should extension dispatch be reactive (World-watch-driven) or imperative?
> 2. How closely does the current `scope.run()` pattern match Vue's `setupStatefulComponent`?

---

## Question 1: Reactive vs Imperative Extension Dispatch

### Current: Imperative Dispatch

```ts
// Framework code explicitly calls this when a node is created
export function dispatchNodeCreated(nodeId: NodeEntityId): void {
  const world = useWorld()
  const comfyClass = world.getComponent(nodeId, NodeType).comfyClass

  for (const ext of nodeExtensions) {
    if (ext.nodeTypes && !ext.nodeTypes.includes(comfyClass)) continue
    ext.nodeCreated?.(createNodeHandle(nodeId))
  }
}
```

Something in the system (a System, the graph loader, the "Add Node" command
handler) must know to call `dispatchNodeCreated()`. This is how the current
v1 extension system works: `comfyApp.ts` manually iterates extensions and
calls hooks at specific points in the execution flow.

### Proposed: Reactive Dispatch (World-Watch-Driven)

```ts
// The extension system WATCHES the World — no manual dispatch needed
watch(
  () => world.queryAll(NodeType),
  (current, previous) => {
    const added = current.filter(id => !previous?.includes(id))
    for (const nodeId of added) mountExtensionsForNode(nodeId)

    const removed = previous?.filter(id => !current.includes(id)) ?? []
    for (const nodeId of removed) unmountExtensionsForNode(nodeId)
  }
)
```

### Decision: **Hybrid — Reactive Mount, Imperative for Non-Entity Hooks**

Pure reactive dispatch is the right model for entity lifecycle hooks
(`nodeCreated`, `loadedGraphNode`, `widgetCreated`) because:

1. **Aligns with ECS philosophy**: Systems react to component changes, not
   imperative calls. The extension system IS a system that reacts to entity
   creation/destruction.
2. **Eliminates coordination burden**: No need to wire up `dispatchNodeCreated()`
   calls in every code path that creates nodes (add node, paste, load workflow,
   undo, CRDT sync). The World is the single source of truth — if a node
   entity exists, extensions mount for it.
3. **Automatic cleanup**: When an entity is removed from the World, the reactive
   watcher detects it and calls `scope.stop()`. No manual `dispatchNodeRemoved`.
4. **Handles edge cases for free**: Batch node creation (paste 50 nodes) just
   creates 50 entities; the watcher picks them all up in the next tick.

But some hooks are NOT entity-bound:
- `init()` / `setup()` — global extension lifecycle
- `beforeConfigureGraph()` / `afterConfigureGraph()` — workflow events
- Keybindings, commands, settings — registered once globally

These remain imperative. The reactive pattern only applies to per-entity hooks.

### Revised Architecture

```ts
// extensionV2Service.ts — the extension mount system

import { EffectScope, watch, onScopeDispose } from 'vue'
import { useWorld } from '@/ecs/world'
import { NodeType } from '@/ecs/components'
import type { NodeEntityId } from '@/ecs/entityIds'

const scopeRegistry = new Map<string, EffectScope>()

/** Called once during app initialization */
export function startExtensionSystem(): void {
  const world = useWorld()

  // Reactive mount — watches World for entity creation/removal
  watch(
    () => world.queryAll(NodeType),
    (currentIds, previousIds) => {
      const prev = new Set(previousIds ?? [])
      const curr = new Set(currentIds)

      // Mount extensions for new nodes
      for (const nodeId of currentIds) {
        if (!prev.has(nodeId)) mountExtensionsForNode(nodeId)
      }

      // Unmount extensions for removed nodes
      for (const id of previousIds ?? []) {
        if (!curr.has(id)) unmountExtensionsForNode(id)
      }
    },
    { flush: 'post' }  // Run after World mutations settle
  )
}

function mountExtensionsForNode(nodeId: NodeEntityId): void {
  const world = useWorld()
  const comfyClass = world.getComponent(nodeId, NodeType).comfyClass

  for (const ext of nodeExtensions) {
    if (ext.nodeTypes && !ext.nodeTypes.includes(comfyClass)) continue
    if (!ext.nodeCreated) continue

    const scope = getOrCreateScope(ext.name, nodeId)
    scope.run(() => {
      ext.nodeCreated!(createNodeHandle(nodeId))
    })
  }
}

function unmountExtensionsForNode(nodeId: NodeEntityId): void {
  for (const ext of nodeExtensions) {
    const key = `${ext.name}:${nodeId}`
    const scope = scopeRegistry.get(key)
    if (scope) {
      scope.stop()  // Stops all watches, runs onScopeDispose callbacks
      scopeRegistry.delete(key)
    }
  }
}
```

### What About `loadedGraphNode`?

`loadedGraphNode` is called for nodes loaded from a saved workflow, as
distinct from newly created nodes. In the reactive model, both appear as
new entities in the World. The distinction is contextual:

**Option A**: A component flag — `world.getComponent(nodeId, NodeType).loadedFromWorkflow`
set by the graph loader. Extensions check this in `nodeCreated`:

```ts
nodeCreated(node) {
  if (node.getProperty('__loadedFromWorkflow')) {
    // Restoration logic
  } else {
    // Fresh creation logic
  }
}
```

**Option B**: Keep `loadedGraphNode` as a separate hook, dispatched reactively
when the World detects entities created with a `LoadedFromWorkflow` tag component.

**Decision**: Option B. Keep separate hooks. The reactive watcher checks for
a tag component:

```ts
function mountExtensionsForNode(nodeId: NodeEntityId): void {
  const world = useWorld()
  const isLoaded = world.hasComponent(nodeId, LoadedFromWorkflow)

  for (const ext of nodeExtensions) {
    if (ext.nodeTypes && !ext.nodeTypes.includes(comfyClass)) continue

    const scope = getOrCreateScope(ext.name, nodeId)
    scope.run(() => {
      const handle = createNodeHandle(nodeId)
      if (isLoaded && ext.loadedGraphNode) {
        ext.loadedGraphNode(handle)
      } else if (!isLoaded && ext.nodeCreated) {
        ext.nodeCreated(handle)
      }
    })
  }
}
```

## Question 2: Scope Pattern Alignment with Vue Core

### Vue's `setupStatefulComponent` Pattern (component.ts:829-927)

```ts
function setupStatefulComponent(instance) {
  // 1. Create proxy context
  instance.proxy = new Proxy(instance.ctx, PublicInstanceProxyHandlers)

  // 2. Pause reactive tracking (setup shouldn't create deps on render)
  pauseTracking()

  // 3. Set current instance AND activate its EffectScope
  const reset = setCurrentInstance(instance)
  //   ↑ This does: internalSetCurrentInstance(instance); instance.scope.on()
  //   scope.on() makes this scope the activeEffectScope

  // 4. Call user's setup() function
  const setupResult = callWithErrorHandling(setup, instance, ...)

  // 5. Reset: restore previous instance and scope
  resetTracking()
  reset()
  //   ↑ This does: instance.scope.off(); internalSetCurrentInstance(prev)
}
```

The critical pattern is:
1. `setCurrentInstance()` — sets a module-global AND activates the scope
2. User code runs — any `watch()`, `computed()`, `onMounted()` auto-register
3. `reset()` — deactivates the scope, restores previous global

### Our Current Pattern (extensionV2Service.ts)

```ts
const scope = getOrCreateScope(ext.name, nodeId)
scope.run(() => {
  ext.nodeCreated!(createNodeHandle(nodeId))
})
```

### Comparison

| Aspect | Vue core | Our pattern | Match? |
|--------|----------|-------------|--------|
| Scope activation | `instance.scope.on()` via `setCurrentInstance()` | `scope.run()` (which sets `activeEffectScope = this`) | ✅ Same |
| Effect capture | `watch/computed` auto-register in `activeEffectScope` | Same — `watch()` inside `scope.run()` registers in scope | ✅ Same |
| Scope cleanup | `scope.stop()` on unmount | `scope.stop()` on entity removal | ✅ Same |
| Context global | `currentInstance` set during setup | **Missing** — no `currentExtension` global | ⚠️ Gap |
| Tracking paused | `pauseTracking()` before setup | Not done | ⚠️ Gap |
| Reset on exit | `reset()` restores previous instance + scope | `scope.run()` auto-restores `activeEffectScope` | ✅ Same |

### Gap: No `currentExtension` Global

Vue uses `currentInstance` so lifecycle hooks (`onMounted`, etc.) can capture
their associated instance without it being passed explicitly. Our system
doesn't have this yet because our hooks are explicit method calls (`ext.nodeCreated(handle)`)
rather than implicit captures (`onMounted(() => ...)`).

**Do we need `currentExtension`?** Not yet. Our hooks pass the handle
directly: `nodeCreated(node)` — the extension always has explicit access to
its context. If we later add implicit hooks like `onNodeRemoved(() => ...)`
that need to capture context, we'd add it then. For now, `onScopeDispose`
from Vue already works inside `scope.run()` without needing a custom global.

### Gap: `pauseTracking()` During Setup

Vue pauses dependency tracking during `setup()` because setup shouldn't
create implicit render deps. In our case, the concern is different: we
don't have a render effect. But we should still pause tracking during
extension setup to prevent the extension's setup from accidentally creating
deps on whatever outer effect might be running:

```ts
scope.run(() => {
  pauseTracking()
  try {
    ext.nodeCreated!(createNodeHandle(nodeId))
  } finally {
    resetTracking()
  }
})
```

This is a minor correctness improvement. Extension setup should declare
explicit subscriptions via `node.on()`, not accidentally track World reads.

### Revised `scope.run` Pattern

```ts
import { pauseTracking, resetTracking } from 'vue'

function mountExtensionForNode(
  ext: NodeExtensionOptions,
  nodeId: NodeEntityId,
  hook: 'nodeCreated' | 'loadedGraphNode'
): void {
  const scope = getOrCreateScope(ext.name, nodeId)
  scope.run(() => {
    pauseTracking()
    try {
      ext[hook]!(createNodeHandle(nodeId))
    } finally {
      resetTracking()
    }
  })
}
```

## Summary

1. **Reactive dispatch** for entity lifecycle hooks (`nodeCreated`,
   `loadedGraphNode`, `widgetCreated`). The extension system watches
   `world.queryAll(NodeType)` and auto-mounts/unmounts extensions. No
   imperative `dispatchNodeCreated()` calls needed.

2. **Imperative dispatch** retained for non-entity hooks (`init`, `setup`,
   workflow events).

3. **Scope pattern matches Vue core** almost exactly. Two minor gaps
   addressed: add `pauseTracking()`/`resetTracking()` around setup, and
   defer `currentExtension` global until implicit hooks are needed.

4. **`loadedGraphNode` distinction** handled via a `LoadedFromWorkflow`
   tag component in the World, checked by the reactive mount system.

### `package.json` requirements

- `"name": "@comfyorg/extension-api"`
- `"version": "0.1.0"` (D6 Phase A: signals experimental)
- `"description"`: one sentence, references docs.comfy.org link
- `"main": "./build/index.js"`
- `"types": "./build/index.d.ts"`
- `"exports"`: object form with `"./*"` subpath exports for granular imports
- `"files": ["build", "README.md", "LICENSE"]`
- `"sideEffects": false`
- `"repository"`, `"homepage"`, `"bugs"` fields pointing at Comfy-Org/ComfyUI_frontend
- `"license": "GPL-3.0-only"` (match parent)
- `"publishConfig": { "access": "public", "provenance": true }`
- `"scripts"`:
  - `"build": "tsc -p tsconfig.build.json"`
  - `"typecheck": "tsc --noEmit"`
  - `"prepublishOnly": "pnpm build"`
- No runtime dependencies. PeerDeps on `vue` (types-only) if needed.
- `"engines"`: `"node": ">=20"`

### `tsconfig.build.json`

Extends root tsconfig. Narrow include to `src/extension-api/`. Set:

- `"declaration": true`
- `"declarationMap": true`
- `"emitDeclarationOnly": false` (we want both `.js` and `.d.ts`)
- `"outDir": "./build"`
- `"rootDir": "../../src/extension-api"`
- `"isolatedModules": true`

## Acceptance criteria

1. `pnpm --filter @comfyorg/extension-api build` produces `build/index.d.ts`
   and `build/index.js`.
2. `pnpm --filter @comfyorg/extension-api typecheck` passes.
3. `cd packages/extension-api && npm pack` produces a tarball.
4. Smoke test: in a temp directory, install the tarball into a fresh
   project (`npm init -y && npm install /path/to/tarball`) and write a
   minimal example extension that typechecks against
   `import { defineExtension } from '@comfyorg/extension-api'`.
5. CI typecheck workflow (PKG4) consumes the build output of this
   package.

## Import-map compatibility requirement

The Notion "Evolution of Frontend Build/Bundling" doc (`research/architecture/notion-build-bundling-and-api-surface.md`) establishes that the canonical long-term path for extensions to consume this package is via **browser import maps**, not relative-path shims:

```html
<!-- index.html (served by ComfyUI backend) -->
<script type="importmap">
{ "imports": { "@comfyorg/extension-api": "/assets/extension-api.mjs" } }
</script>
```

```js
// extension code — no relative path, no window.comfyAPI
import { defineExtension } from '@comfyorg/extension-api'
```

**PKG3 build must emit a single ESM bundle** (`build/index.mjs`) alongside the standard `build/index.js` CJS output, suitable for serving as the import-map target. The `package.json` `"exports"` field should include:

```json
"exports": {
  ".": {
    "import": "./build/index.mjs",
    "require": "./build/index.js",
    "types": "./build/index.d.ts"
  }
}
```

The shim plugin (`build/plugins/comfyAPIPlugin.ts` in ComfyUI_frontend) will need updating during D6 Phase B/C to emit shims for `@comfyorg/extension-api` exports in addition to the existing `window.comfyAPI.*` bindings — but that is out of scope for PKG3 and tracked in MIG1.

**Do NOT** add `window.comfyAPI.*` bindings inside PKG3 itself. That is the shim plugin's responsibility. PKG3 is a pure ESM package; it has no side effects on `window`.
